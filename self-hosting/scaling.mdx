## Overview

This guide covers performance optimization strategies for scaling Mempool from a basic single-server setup to a high-performance cluster capable of handling significant traffic.

<Info>
  These optimizations are primarily for **advanced users** running public instances or experiencing performance issues.
</Info>

## Backend Optimization

### Process Clustering

Mempool backend can spawn multiple processes to handle concurrent requests:

```json mempool-config.json
{
  "MEMPOOL": {
    "SPAWN_CLUSTER_PROCS": 0,  // 0 = auto (number of CPU cores)
    // Or set manually:
    // "SPAWN_CLUSTER_PROCS": 4,  // 4 processes
  }
}
```

<Tabs>
  <Tab title="Production (Auto)">
    ```json
    "SPAWN_CLUSTER_PROCS": 0  // Recommended: auto-detect cores
    ```
    
    Uses all available CPU cores for maximum throughput
  </Tab>
  
  <Tab title="Manual Control">
    ```json  
    "SPAWN_CLUSTER_PROCS": 4  // Fixed 4 processes
    ```
    
    Set manually when running alongside other services
  </Tab>
  
  <Tab title="Development">
    ```json
    "SPAWN_CLUSTER_PROCS": 0  // Single process
    ```
    
    Disabled for development (default in Docker)
  </Tab>
</Tabs>

### Polling Rate Optimization

Adjust how frequently Mempool polls Bitcoin Core:

```json mempool-config.json
{
  "MEMPOOL": {
    "POLL_RATE_MS": 1000,  // Production: 1000ms (1 second)
    // High traffic: 500
    // Low resources: 2000
  }
}
```

<Warning>
  Lower poll rates increase CPU usage. Balance responsiveness with system load.
</Warning>

### Database Performance

Optimize MariaDB for Mempool workloads:

```cnf my.cnf / mariadb.conf
[mysqld]
# Connection settings
max_connections = 200
connect_timeout = 10

# Buffer pool (set to 50-70% of available RAM)
innodb_buffer_pool_size = 8G
innodb_buffer_pool_instances = 8

# Log file size
innodb_log_file_size = 512M
innodb_log_buffer_size = 64M

# Flush behavior (balance durability vs speed)
innodb_flush_log_at_trx_commit = 2
innodb_flush_method = O_DIRECT

# Query cache (deprecated in MariaDB 10.5+, use InnoDB buffer)
query_cache_type = 0

# Table cache
table_open_cache = 4096
table_definition_cache = 2048

# Temporary tables
tmp_table_size = 256M
max_heap_table_size = 256M

# Thread handling
thread_cache_size = 128

# Logging
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 2
```

<Steps>
  <Step title="Calculate Buffer Pool Size">
    ```bash
    # Use 50-70% of RAM for buffer pool
    # Example for 16GB RAM system:
    innodb_buffer_pool_size = 8G
    ```
  </Step>
  
  <Step title="Apply Configuration">
    ```bash
    # Restart MariaDB
    sudo systemctl restart mariadb
    
    # Verify settings
    mysql -e "SHOW VARIABLES LIKE 'innodb_buffer_pool_size';"
    ```
  </Step>
  
  <Step title="Monitor Performance">
    ```sql
    -- Check buffer pool usage
    SHOW ENGINE INNODB STATUS\G
    
    -- View slow queries
    SELECT * FROM mysql.slow_log ORDER BY start_time DESC LIMIT 10;
    ```
  </Step>
</Steps>

## Redis Caching

Redis dramatically improves performance by caching frequently accessed data.

### Installation & Configuration

<Tabs>
  <Tab title="Debian/Ubuntu">
    ```bash
    sudo apt-get update
    sudo apt-get install redis-server
    sudo systemctl enable redis-server
    sudo systemctl start redis-server
    ```
  </Tab>
  
  <Tab title="FreeBSD">
    ```bash
    pkg install redis
    sysrc redis_enable="YES"
    service redis start
    ```
  </Tab>
  
  <Tab title="Docker">
    ```yaml docker-compose.yml
    services:
      redis:
        image: redis:7-alpine
        restart: on-failure
        volumes:
          - ./redis-data:/data
        command: redis-server --save 60 1 --loglevel warning
    ```
  </Tab>
</Tabs>

### Redis Configuration

```conf redis.conf
# Unix socket (faster than TCP)
unixsocket /tmp/redis.sock
unixsocketperm 666

# Also listen on TCP for monitoring
bind 127.0.0.1 ::1
port 6379

# Memory management
maxmemory 2gb
maxmemory-policy allkeys-lru

# Persistence (balance durability vs performance)
save 3600 1 300 100 60 10000
rdbcompression yes
rdbchecksum yes

# Performance
tcp-backlog 511
tcp-keepalive 300
timeout 0

# Logging
loglevel notice
logfile /var/log/redis/redis.log

# Disable slow features
appendonly no
```

### Mempool Redis Configuration

```json mempool-config.json
{
  "REDIS": {
    "ENABLED": true,
    "UNIX_SOCKET_PATH": "/tmp/redis.sock",
    // Or use TCP:
    // "HOST": "127.0.0.1",
    // "PORT": 6379,
    "BATCH_QUERY_BASE_SIZE": 5000
  }
}
```

<Info>
  Unix sockets are ~20% faster than TCP for local Redis connections.
</Info>

### Monitoring Redis

```bash Redis Commands
# Connect to Redis
redis-cli

# Check stats
INFO stats
INFO memory

# Monitor commands in real-time
MONITOR

# View cache hit rate
INFO stats | grep -E 'keyspace_hits|keyspace_misses'

# Check memory usage
INFO memory | grep used_memory_human

# List keys (don't run in production!)
KEYS mempool:*

# Clear cache if needed
FLUSHDB
```

## Nginx Caching

Nginx caching reduces backend load by serving cached responses.

### Cache Configuration

```nginx nginx.conf
http {
    # Define cache zones
    proxy_cache_path /var/cache/nginx/mempool
        levels=1:2
        keys_zone=mempool_cache:20m
        max_size=500m
        inactive=600s
        use_temp_path=off;
    
    # Cache for API responses
    proxy_cache_path /var/cache/nginx/api
        levels=1:2  
        keys_zone=api_cache:100m
        max_size=2g
        inactive=3600s
        use_temp_path=off;
    
    # Cache settings
    proxy_cache_valid 200 10m;
    proxy_cache_valid 404 1m;
    proxy_cache_use_stale error timeout updating http_500 http_502 http_503 http_504;
    proxy_cache_background_update on;
    proxy_cache_lock on;
    
    # Cache control headers
    proxy_ignore_headers Cache-Control;
    proxy_hide_header Set-Cookie;
    
    server {
        listen 80;
        server_name mempool.example.com;
        
        # Static assets (aggressive caching)
        location ~* \.(js|css|png|jpg|jpeg|gif|svg|ico|woff|woff2|ttf|eot)$ {
            root /var/www/mempool/browser;
            expires 1y;
            add_header Cache-Control "public, immutable";
            access_log off;
        }
        
        # API endpoints (selective caching)
        location /api/v1/ {
            proxy_cache api_cache;
            proxy_cache_key $uri$is_args$args;
            
            # Cache certain endpoints longer
            location ~ ^/api/v1/(blocks|statistics|mining) {
                proxy_cache_valid 200 5m;
            }
            
            # Don't cache real-time data
            location ~ ^/api/v1/(mempool|transactions/cpfp) {
                proxy_cache off;
            }
            
            proxy_pass http://mempool_backend;
            
            # Add cache status header
            add_header X-Cache-Status $upstream_cache_status;
        }
        
        # SSR pages
        location / {
            proxy_cache mempool_cache;
            proxy_cache_key $uri$is_args$args;
            proxy_cache_valid 200 10m;
            
            proxy_pass http://mempool_backend;
        }
    }
    
    # Backend upstream
    upstream mempool_backend {
        server 127.0.0.1:8999;
        keepalive 32;
    }
}
```

### Cache Warming

Pre-populate cache with frequently accessed content:

```bash nginx-cache-warmer.sh
#!/bin/bash
# Warm nginx cache by requesting popular endpoints

HOST="https://mempool.example.com"
DELAY=0.15

# Popular URLs to cache
URLs=(
    "/"
    "/mining"
    "/lightning"  
    "/graphs"
    "/api/v1/blocks"
    "/api/v1/statistics/24h"
    "/api/v1/statistics/1w"
    "/api/v1/statistics/1m"
    "/api/v1/mining/pools/24h"
    "/api/v1/mining/pools/1w"
    "/api/v1/mining/pools/1m"
    "/api/v1/mining/hashrate/1m"
    "/api/v1/mining/hashrate/3m"
    "/api/v1/mining/reward-stats/144"
)

for url in "${URLs[@]}"; do
    echo "Warming: $HOST$url"
    curl -s -o /dev/null "$HOST$url"
    sleep $DELAY
done

echo "Cache warming complete"
```

Run via cron:

```cron
# Warm cache every 5 minutes
*/5 * * * * /usr/local/bin/nginx-cache-warmer.sh
```

<Tip>
  Cache warming is especially useful after deployments or cache flushes.
</Tip>

## Bitcoin Core Optimization

### Configuration Tuning

```bash bitcoin.conf
# Database cache (increase during IBD, reduce after)
dbcache=4096  # 4GB - use 50% of RAM during sync

# After sync complete, reduce to:
# dbcache=1024

# Mempool settings
maxmempool=4096      # 4GB mempool
mempoolexpiry=999999 # Never expire (for statistics)

# Indexing (required for Mempool)
txindex=1
coinstatsindex=1

# Network performance
maxconnections=100
maxuploadtarget=0    # Unlimited upload

# Script verification threads
par=8  # Use 8 cores for signature verification

# RPC optimizations
rpcthreads=16
rpcworkqueue=256

# ZMQ for real-time updates (optional)
zmqpubrawblock=tcp://127.0.0.1:8334
zmqpubrawtx=tcp://127.0.0.1:8335
```

### Multiple Bitcoin Nodes

For high availability, run multiple Bitcoin nodes:

```json mempool-config.json
{
  "CORE_RPC": {
    "HOST": "127.0.0.1",
    "PORT": 8332,
    "USERNAME": "mempool",
    "PASSWORD": "password1"
  },
  "SECOND_CORE_RPC": {
    "HOST": "127.0.0.1",
    "PORT": 8302,
    "USERNAME": "mempool",  
    "PASSWORD": "password2"
  },
  "MEMPOOL": {
    "USE_SECOND_NODE_FOR_MINFEE": true
  }
}
```

<Info>
  Using a second node reduces load and provides redundancy.

### Configuration

```toml electrs.toml
# Network
network = "bitcoin"

# Performance
```

### Memory Recommendations

| RAM Available | Optimal Settings |
|---------------|------------------|
| 8GB | `index_batch_size=50`, `bulk_index_threads=2` |
| 16GB | `index_batch_size=100`, `bulk_index_threads=4` |
| 32GB | `index_batch_size=200`, `bulk_index_threads=8` |
| 64GB+ | `index_batch_size=500`, `bulk_index_threads=16` |

## Network & Rate Limiting

### Nginx Rate Limiting

```nginx nginx.conf
http {
    # Define rate limit zones
    limit_req_zone $binary_remote_addr zone=api:10m rate=200r/m;
    limit_req_zone $binary_remote_addr zone=electrs:10m rate=2000r/m;
    limit_conn_zone $binary_remote_addr zone=websocket:10m;
    limit_req_status 429;
    
    # Exempt localhost
    geo $limited_ip {
        default         1;
        127.0.0.1       0;
        192.168.0.0/16  0;  # Local network
    }
    
    map $limited_ip $limited_ip_key {
        1 $binary_remote_addr;
        0 '';
    }
    
    server {
        # Apply rate limits
        location /api/ {
            limit_req zone=api burst=20 nodelay;
            limit_req_status 429;
            
            proxy_pass http://backend;
        }
        
        # WebSocket rate limiting
        location /ws {
            limit_conn websocket 10;  # Max 10 connections per IP
            
            proxy_pass http://backend;
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection "upgrade";
        }
    }
}
```

### Connection Limits

```nginx nginx.conf
# Worker configuration
worker_processes auto;
worker_rlimit_nofile 100000;

events {
    worker_connections 9000;
    multi_accept on;
    use epoll;  # Linux
}

http {
    # Timeout settings
    client_body_timeout 10s;
    client_header_timeout 10s;
    keepalive_timeout 69s;
    send_timeout 69s;
    
    # Keep-alive
    keepalive_requests 1337;
    
    # Reset timed out connections
    reset_timedout_connection on;
}
```

## Load Balancing

For extremely high traffic, run multiple Mempool backends:

```nginx nginx.conf
upstream mempool_cluster {
    least_conn;  # Load balancing method
    
    server 127.0.0.1:8999 weight=2;
    server 127.0.0.1:9000 weight=2;
    server 127.0.0.1:9001 weight=1;
    server 127.0.0.1:9002 weight=1 backup;
    
    keepalive 64;
}

server {
    location /api/ {
        proxy_pass http://mempool_cluster;
        proxy_next_upstream error timeout invalid_header http_500;
        proxy_connect_timeout 2s;
    }
}
```

Run multiple backend instances:

```bash
# Instance 1
HTTP_PORT=8999 npm run start

# Instance 2  
HTTP_PORT=9000 npm run start

# Instance 3
HTTP_PORT=9001 npm run start
```

## CDN Integration

For global performance, use a CDN:

<Steps>
  <Step title="Choose CDN Provider">
    Popular options:
    - Cloudflare (free tier available)
    - AWS CloudFront
    - Fastly
    - BunnyCDN
  </Step>
  
  <Step title="Configure Cache Rules">
    ```javascript Cloudflare Cache Rules
    // Cache static assets aggressively
    if (request.url.pathname.match(/\.(js|css|png|jpg|svg|woff2?)$/)) {
      return {
        cacheEverything: true,
        edgeTTL: 31536000,  // 1 year
        browserTTL: 31536000
      };
    }
    
    // Cache API selectively
    if (request.url.pathname.startsWith('/api/v1/statistics')) {
      return {
        cacheEverything: true,
        edgeTTL: 300,  // 5 minutes
        browserTTL: 60
      };
    }
    
    // Don't cache real-time data
    if (request.url.pathname.startsWith('/api/v1/mempool')) {
      return { cache: false };
    }
    ```
  </Step>
  
  <Step title="Configure Origin">
    Point CDN to your Mempool instance origin
  </Step>
</Steps>

## Troubleshooting Performance

<AccordionGroup>
  <Accordion title="High CPU usage">
    **Causes:**
    - Too many cluster processes
    - Bitcoin Core signature verification
    - Electrs indexing
    
    **Solutions:**
    - Reduce `SPAWN_CLUSTER_PROCS`
    - Adjust Bitcoin `par` setting
    - Limit concurrent connections
  </Accordion>
  
  <Accordion title="High memory usage">
    **Causes:**
    - Large MariaDB buffer pool
    - Redis cache too large
    - Bitcoin dbcache setting
    
    **Solutions:**
    - Tune `innodb_buffer_pool_size`
    - Set Redis `maxmemory`
    - Reduce Bitcoin `dbcache` after sync
  </Accordion>
  
  <Accordion title="Slow API responses">
    **Causes:**
    - No caching enabled
    - Database not optimized
    - Slow disk I/O
    
    **Solutions:**
    - Enable Redis
    - Configure nginx caching
    - Optimize MariaDB
    - Upgrade to NVMe SSD
  </Accordion>
  
  <Accordion title="WebSocket disconnections">
    **Causes:**
    - Nginx timeout too short
    - Too many connections
    - Rate limiting
    
    **Solutions:**
    ```nginx
    location /ws {
        proxy_read_timeout 3600s;
        proxy_send_timeout 3600s;
        limit_conn websocket 50;
    }
    ```
  </Accordion>
</AccordionGroup>
