## Before You Start

All contributors must agree to the **Contributor License Agreement (CLA)** before their contributions can be merged.

<Warning>
  First-time contributors must create a CLA agreement file before submitting pull requests.
</Warning>

## Quick Start

<Steps>
  <Step title="Fork and Clone">
    Fork the repository and clone it locally:

    ```bash
    git clone https://github.com/YOUR_USERNAME/mempool.git
    cd mempool
    ```
  </Step>

  <Step title="Create a Branch">
    Create a feature branch for your changes:

    ```bash
    git checkout -b feature/your-feature-name
    ```
  </Step>

  <Step title="Make Your Changes">
    Make your changes following the [code standards](#code-standards) below.
  </Step>

  <Step title="Sign Your Commits">
    <Note>
      All commits **must** be GPG-signed.
    </Note>

    Enable GPG signing:

    ```bash
    git config commit.gpgsign true
    ```

    Commit your changes:

    ```bash
    git add .
    git commit -m "feat: add new feature"
    ```
  </Step>

  <Step title="Run Tests">
    Ensure all tests pass:

    ```bash
    # Backend tests
    cd backend
    npm test

    # Frontend tests
    cd ../frontend
    npm test
    npm run cypress:run
    ```
  </Step>

  <Step title="Push and Create PR">
    Push your branch and create a pull request:

    ```bash
    git push origin feature/your-feature-name
    ```

    Go to GitHub and open a pull request against the `master` branch.
  </Step>
</Steps>

## Contributor License Agreement

### First-Time Contributors

When submitting your first pull request, you must create a CLA agreement file:

<Steps>
  <Step title="Create Your CLA File">
    Create a file named `/contributors/{github_username}.txt` in the repository.
  </Step>

  <Step title="Add Agreement Statement">
    In this file, indicate your agreement to the CLA terms:

    ```text
    I, [Your Full Name], agree to the terms of the Contributor License Agreement
    as outlined in CONTRIBUTING.md, dated [Current Date].

    GitHub Username: @your_username
    Date: YYYY-MM-DD
    ```
  </Step>

  <Step title="Include in Pull Request">
    Include this file in your pull request along with your code changes.
  </Step>
</Steps>

<Tip>
  See existing contributor files in the `/contributors/` directory for examples.
</Tip>

## Code Standards

### TypeScript Style

Both frontend and backend use TypeScript with ESLint for code quality.

<Tabs>
  <Tab title="Backend">
    ```bash
    cd backend

    # Check for linting errors
    npm run lint

    # Auto-fix linting issues
    npm run lint:fix

    # Format code with Prettier
    npm run prettier
    ```
  </Tab>

  <Tab title="Frontend">
    ```bash
    cd frontend

    # Check for linting errors
    npm run lint

    # Auto-fix linting issues
    npm run lint:fix

    # Format code with Prettier
    npm run prettier
    ```
  </Tab>
</Tabs>

### Commit Message Convention

Use conventional commit messages:

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Types:**
- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation changes
- `style`: Code style changes (formatting, etc.)
- `refactor`: Code refactoring
- `test`: Adding or updating tests
- `chore`: Maintenance tasks

**Examples:**

<CodeGroup>
```bash Feature
feat(mining): add support for new mining pool detection

Implements automatic detection of XYZ mining pool using
coinbase transaction patterns.

Closes #123
```

```bash Bug Fix
fix(websocket): resolve memory leak in connection handler

Properly clean up event listeners when connections close.

Fixes #456
```

```bash Documentation
docs(api): update REST API endpoint documentation

Add examples for fee estimation endpoints.
```
</CodeGroup>

### GPG Signing

<Warning>
  All commits must be GPG-signed. Pull requests with unsigned commits will not be merged.
</Warning>

<Steps>
  <Step title="Generate GPG Key">
    If you don't have a GPG key:

    ```bash
    gpg --full-generate-key
    ```

    Choose:
    - Key type: RSA and RSA
    - Key size: 4096 bits
    - Expiration: Your preference
  </Step>

  <Step title="Add Key to GitHub">
    Export your public key:

    ```bash
    gpg --armor --export YOUR_EMAIL
    ```

    Add it to GitHub: Settings → SSH and GPG keys → New GPG key
  </Step>

  <Step title="Configure Git">
    ```bash
    git config --global user.signingkey YOUR_KEY_ID
    git config --global commit.gpgsign true
    ```
  </Step>
</Steps>

## Testing Requirements

All contributions must include appropriate tests.

### Backend Tests

<Tabs>
  <Tab title="Unit Tests">
    Use Jest for unit tests:

    ```typescript
    // backend/src/__tests__/example.test.ts
    describe('MyFunction', () => {
      it('should return expected result', () => {
        expect(myFunction(input)).toBe(expectedOutput);
      });
    });
    ```

    Run tests:
    ```bash
    npm test
    ```
  </Tab>

  <Tab title="Integration Tests">
    Integration tests require a database:

    ```bash
    npm run test:integration
    ```
  </Tab>

  <Tab title="Coverage">
    Ensure adequate test coverage:

    ```bash
    npm run test:ci
    ```

    Coverage reports are generated in `coverage/`.
  </Tab>
</Tabs>

### Frontend Tests

<Tabs>
  <Tab title="Unit Tests">
    Angular unit tests:

    ```bash
    npm test
    ```
  </Tab>

  <Tab title="E2E Tests">
    Cypress end-to-end tests:

    ```bash
    # Interactive mode
    npm run cypress:open

    # Headless mode
    npm run cypress:run
    ```
  </Tab>
</Tabs>

<Note>
  See the [Testing Guide](/development/testing) for detailed testing information.
</Note>

## Pull Request Process

### Before Submitting

<Checklist>
  - [ ] Code follows style guidelines
  - [ ] All commits are GPG-signed
  - [ ] CLA agreement file created (first-time contributors)
  - [ ] Tests added/updated and passing
  - [ ] Documentation updated (if applicable)
  - [ ] Commit messages follow convention
  - [ ] Branch is up to date with master
</Checklist>

### PR Description Template

Include in your pull request:

```markdown
## Description
Brief description of changes

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change
- [ ] Documentation update

## Testing
Describe testing performed

## Screenshots (if applicable)
Add screenshots for UI changes

## Checklist
- [ ] Tests pass locally
- [ ] Commits are GPG-signed
- [ ] CLA agreement included (if first PR)
```

### Review Process

<Steps>
  <Step title="Automated Checks">
    CI/CD runs automatically:
    - Linting
    - Type checking
    - Unit tests
    - Integration tests
    - E2E tests (frontend)
  </Step>

  <Step title="Code Review">
    Team members review your code:
    - Code quality
    - Test coverage
    - Design decisions
    - Documentation
  </Step>

  <Step title="Address Feedback">
    Make requested changes:
    ```bash
    git add .
    git commit -m "fix: address review feedback"
    git push
    ```
  </Step>

  <Step title="Merge">
    Once approved, a maintainer will merge your PR.
  </Step>
</Steps>
