Mempool is a full-stack Bitcoin blockchain explorer and mempool visualizer built with modern web technologies. This guide provides an overview of the system architecture, components, and data flow.

## Overview

Mempool follows a client-server architecture with real-time data synchronization via WebSockets.

## Backend Architecture

The Mempool backend is built with **Node.js**, **Express**, and **TypeScript**, providing a robust API and real-time data processing.

### Core Components

<AccordionGroup>
  <Accordion title="API Layer" icon="route">
    The REST API is organized into domain-specific routes:

    - **Bitcoin Routes** (`backend/src/api/bitcoin/bitcoin.routes.ts`): Core blockchain data
    - **Mining Routes** (`backend/src/api/mining/mining-routes.ts`): Mining statistics and pools
    - **Lightning Routes** (`backend/src/api/explorer/`): Lightning Network data
    - **Statistics Routes** (`backend/src/api/statistics/statistics.routes.ts`): Historical stats
    - **Liquid Routes** (`backend/src/api/liquid/liquid.routes.ts`): Liquid sidechain data
    - **Acceleration Routes** (`backend/src/api/acceleration/`): Transaction acceleration services

    All routes are mounted in `backend/src/index.ts`.
  </Accordion>

  <Accordion title="WebSocket Handler" icon="tower-broadcast">
    Real-time updates are delivered via WebSocket connections:

    **File**: `backend/src/api/websocket-handler.ts`

    ```typescript
    // WebSocket message types
    - 'want': Subscribe to specific data streams
    - 'init': Initial connection data
    - 'mempool-blocks': Projected mempool blocks
    - 'live-2h-chart': Real-time fee chart data
    - 'stats': Mempool statistics
    - 'block': New block notifications
    ```

    Clients subscribe to specific data streams and receive updates as blockchain state changes.
  </Accordion>

  <Accordion title="Mempool Engine" icon="memory">
    The core mempool processing engine tracks and analyzes unconfirmed transactions:

    **File**: `backend/src/api/mempool.ts`

    Key responsibilities:
    - Parse mempool transactions from Bitcoin Core
    - Calculate effective fee rates
    - Build mempool blocks (fee-based transaction clusters)
    - Track transaction dependencies
    - Update statistics and charts

    Updates run continuously in a polling loop.
  </Accordion>

  <Accordion title="Blocks Module" icon="cube">
    Handles block processing and historical data:

    **File**: `backend/src/api/blocks.ts`

    - Fetches new blocks from Bitcoin Core
    - Calculates block statistics (fees, transactions, weight)
    - Identifies mining pools
    - Stores block data in database
    - Manages blockchain reorganizations
  </Accordion>

  <Accordion title="Indexer" icon="database">
    Background service for indexing blockchain data:

    **File**: `backend/src/indexer.ts`

    - Re-indexes historical blocks
    - Updates mining pool assignments
    - Rebuilds statistics tables
    - Handles data migrations
  </Accordion>

  <Accordion title="Statistics Engine" icon="chart-line">
    Aggregates and processes blockchain statistics:

    **File**: `backend/src/api/statistics/statistics.ts`

    - Fee rate percentiles
    - Transaction volume
    - Mempool size over time
    - Mining statistics
    - Network difficulty and hashrate
  </Accordion>
</AccordionGroup>

## Data Flow

Understanding how data flows through the system:

<Steps>
  <Step title="Bitcoin Core Sync">
    The backend polls Bitcoin Core via RPC:

    ```
    Backend → Bitcoin Core RPC
      ├─ getmempoolinfo()
      ├─ getrawmempool(true)
      ├─ getblock(hash)
      └─ getrawtransaction(txid)
    ```
  </Step>

  <Step title="Data Processing">
    Raw data is processed and enriched:

    - Calculate effective fee rates
    - Identify mining pools
    - Build mempool blocks
    - Compute statistics
  </Step>

  <Step title="Database Storage">
    Processed data is stored in MariaDB:

    ```
    Repositories → Database
      ├─ BlocksRepository
      ├─ TransactionsRepository
      ├─ PoolsRepository
      └─ StatisticsRepository
    ```
  </Step>

  <Step title="API & WebSocket">
    Data is exposed to clients:

    ```
    Frontend ← WebSocket ← Backend
    Frontend ← REST API ← Backend
    ```
  </Step>

  <Step title="Frontend Rendering">
    Angular components render the data:

    ```
    Components subscribe to:
      - WebSocket streams (real-time)
      - HTTP endpoints (on-demand)
    ```
  </Step>
</Steps>

## Performance Optimizations

### Backend

<AccordionGroup>
  <Accordion title="Cluster Mode">
    Scale horizontally with multiple worker processes:

    ```json
    {
      "MEMPOOL": {
        "SPAWN_CLUSTER_PROCS": 4
      }
    }
    ```

    See: `backend/src/index.ts:70-96`
  </Accordion>

  <Accordion title="Disk Cache">
    Frequently accessed data is cached to disk:

    ```typescript
    // backend/src/api/disk-cache.ts
    - Block data
    - Transaction details
    - Address histories
    ```
  </Accordion>

  <Accordion title="Redis Cache">
    Optional Redis integration for distributed caching:

    ```typescript
    // backend/src/api/redis-cache.ts
    - API responses
    - Rate limiting counters
    ```
  </Accordion>

  <Accordion title="Database Indexing">
    Optimized indexes on frequently queried columns:

    - `blocks.height`
    - `blocks.hash`
    - `transactions.txid`
    - `statistics.added`
  </Accordion>
</AccordionGroup>
